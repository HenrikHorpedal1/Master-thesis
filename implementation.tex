\section{Software}
Only the modules that have been developed or significantly changed since the start of the thesis will be presented.
The firmware targets an ESP32-S3 running the Arduino framework on top of FreeRTOS.
All inter-task communication uses FreeRTOS queues and spinlock-protected shared state,
so no module blocks another.

\subsection{Main}
The entry point creates two FreeRTOS queues and then spawns all application tasks
before yielding permanently in \texttt{loop()}.
The input queue carries \texttt{InputEvent} structs (footswitch transitions, expression-pedal readings, MIDI notes and CC values);
the clock queue carries \texttt{ClockMsg} structs (MIDI realtime ticks and transport messages).
Separating the two queues keeps timing-critical clock messages from being delayed by larger input events.

Task priorities are assigned so that time-sensitive work always pre-empts lower-priority work:
the controller task (priority 5) pre-empts the clock-sync and input-handler tasks (priority 4),
which in turn pre-empt the MIDI listener and footswitch tasks (priority 3).
The mode-selector task sits at the lowest priority (2) because it only needs to poll a rotary switch at 5~Hz.
\autoref{lst:main-setup} shows the queue and task creation in \texttt{setup()}.

\begin{listing}[H]
	\begin{minted}{cpp}
g_inputQueue = xQueueCreate(16, sizeof(InputEvent));
g_clockQueue = xQueueCreate(32, sizeof(ClockMsg));

g_midiParams.inputQueue = g_inputQueue;
g_midiParams.clockQueue = g_clockQueue;

xTaskCreate(footSwitchTask,   "footSwitchTask",   4096, g_inputQueue,   3, nullptr);
xTaskCreate(midiListnerTask,  "midiListnerTask",  4096, &g_midiParams,  3, nullptr);
xTaskCreate(clockSyncTask,    "clockSyncTask",    4096, g_clockQueue,   4, nullptr);
xTaskCreate(inputHandlerTask, "inputHandlerTask", 4096, g_inputQueue,   4, nullptr);
xTaskCreate(controllerTask,   "ControllerTask",   4096, nullptr,        5, nullptr);
xTaskCreate(modeSelectorTask, "modeSelectorTask", 2048, nullptr,        2, nullptr);
\end{minted}
	\caption{Queue creation and task spawning in \texttt{setup()}.}
	\label{lst:main-setup}
\end{listing}

\subsection{Clock-sync}
\texttt{clockSyncTask} receives \texttt{ClockMsg} events from the clock queue and maintains
a phase-locked estimate of the incoming MIDI clock.
The message type and its payload are defined in \autoref{lst:clockmsg}.

\begin{listing}[H]
	\begin{minted}{cpp}
enum class ClockMsgType : uint8_t {
    Tick, Start, Stop, ContinueMsg, SongPosition
};

struct ClockMsg {
    ClockMsgType type;
    int64_t      t_us;   // hardware timestamp from esp_timer_get_time()
    uint16_t     spp;    // song position pointer (SongPosition messages only)
};
\end{minted}
	\caption{\texttt{ClockMsg} type definitions from \texttt{clock\_sync.h}.}
	\label{lst:clockmsg}
\end{listing}

MIDI specifies 24 timing clock messages per quarter note, so the tick period at 120~BPM is
\begin{equation}
	T_\text{tick} = \frac{60 \times 10^6\,\mu\text{s}}{120 \times 24} \approx 20833\,\mu\text{s}.
\end{equation}
Each tick is timestamped with \texttt{esp\_timer\_get\_time()} in the MIDI callback immediately
and placed on the queue, preserving the hardware timestamp across the scheduling boundary.

The lock-on algorithm is a discrete-time PI loop implemented in \texttt{processTick()}.
On the first tick after a Start message the measured inter-tick interval is used directly
to seed the period estimate $T$.
On subsequent ticks the phase error $e_k$ is the difference between the observed arrival time
and the predicted arrival time:
\begin{equation}
	e_k = t_k - \hat{t}_k .
\end{equation}
The period and the next prediction are then corrected by a proportional--integral adjustment,
\begin{equation}
	\Delta_k = -\!\left(K_P \, e_k + K_I \textstyle\sum_j e_j\right), \qquad
	T_{k+1} = T_k + \Delta_k, \qquad
	\hat{t}_{k+1} = t_k + T_{k+1} + \Delta_k,
\end{equation}
with $K_P = 0.20$ and $K_I = 0.02$.
Errors larger than half the current period are clamped to suppress USB-jitter outliers,
and the phase adjustment is further clamped to $\pm 30\%$ of the period to prevent
the estimate from diverging on a single bad tick.
\autoref{alg:processtick} summarises the full algorithm.

\begin{algorithm}[H]
	\caption{\texttt{processTick}($t$) --- PI clock recovery}
	\label{alg:processtick}
	\begin{algorithmic}[1]
		\If{not running} \Return \EndIf
		\If{not locked}
		\State $\mathit{dt} \gets t - t_\text{last}$
		\If{$\mathit{dt} \notin [1000,\; 200000]\;\mu\text{s}$} \Return \EndIf
		\State $T \gets \mathit{dt}$;\quad $\hat{t}_\text{next} \gets t + T$;\quad $\text{locked} \gets \text{true}$
		\State \Return
		\EndIf
		\State $e \gets t - \hat{t}_\text{next}$ \Comment{phase error}
		\State $e \gets \mathrm{clamp}(e,\; {-0.5T},\; {+0.5T})$ \Comment{reject outliers}
		\State $\Delta \gets -(K_P \cdot e + K_I \cdot \Sigma e)$
		\State $\Delta \gets \mathrm{clamp}(\Delta,\; {-0.3T},\; {+0.3T})$
		\State $T \gets T + \Delta$ \Comment{frequency correction}
		\State $t_\text{last} \gets t$;\quad $\hat{t}_\text{next} \gets t + T + \Delta$ \Comment{phase correction}
		\State $n_\text{tick} \gets n_\text{tick} + 1$
	\end{algorithmic}
\end{algorithm}

Thread-safe getters (\texttt{clockSyncGetBpm()}, \texttt{clockSyncGetTickCount()}, \texttt{clockSyncGetPhase()})
allow other tasks to read the smoothed BPM, the cumulative tick counter, and the fractional
phase within the current tick without holding the spinlock for longer than necessary.
\texttt{clockSyncGetPhase()} interpolates the phase between ticks in real time using
\texttt{esp\_timer\_get\_time()}, giving sub-tick resolution to the controller.
If no tick arrives for more than 500~ms the module declares the clock lost and clears the lock flag.

\subsection{Moteus-config}
\texttt{moteus-config} encapsulates the CAN-FD peripheral and exposes a single \texttt{hornMoteus()} singleton.
The ACAN2517FD controller is initialised over SPI at 10~MHz, with the CAN bit rate set to 1~Mbit/s.
Bit-rate switching (BRS) is disabled because the cable run is short and the simpler classical-CAN framing
reduces the risk of reflections at the chosen baud rate.

At startup, base PID gains are written to the Moteus controller via its text diagnostic interface,
as shown in \autoref{lst:moteus-pid}.
These values are stored only in RAM and apply to both position and velocity closed-loop modes.

\begin{listing}[H]
	\begin{minted}{cpp}
// Base gains written to Moteus RAM at startup
static constexpr double MOTEUS_BASE_KP = 7.0;
static constexpr double MOTEUS_BASE_KI = 0.4;
static constexpr double MOTEUS_BASE_KD = 0.7;

snprintf(buf, sizeof(buf), "conf set servo.pid_position.kp %.4f", MOTEUS_BASE_KP);
g_horn.DiagnosticCommand(buf); delay(20);
snprintf(buf, sizeof(buf), "conf set servo.pid_position.ki %.4f", MOTEUS_BASE_KI);
g_horn.DiagnosticCommand(buf); delay(20);
snprintf(buf, sizeof(buf), "conf set servo.pid_position.kd %.4f", MOTEUS_BASE_KD);
g_horn.DiagnosticCommand(buf); delay(20);
\end{minted}
	\caption{Base PID gain configuration sent to the Moteus at startup via diagnostic commands.}
	\label{lst:moteus-pid}
\end{listing}

The controller task later scales these gains at run-time using the Moteus
\texttt{kp\_scale}, \texttt{kd\_scale}, and \texttt{ilimit\_scale} fields,
which are transmitted every cycle so the effective gains can change without a reconfiguration round-trip.

\subsection{Controller}
\texttt{controllerTask} runs at 100~Hz and translates the current \texttt{ReferenceState}
(desired speed in RPM) into Moteus \texttt{SetPosition} commands.
Three internal control modes are defined as the \texttt{ControlMode} enum shown in \autoref{lst:controlmode}.

\begin{listing}[H]
	\begin{minted}{cpp}
enum class ControlMode { Velocity, BeatSync, Position };
\end{minted}
	\caption{\texttt{ControlMode} enum defining the three operating states of the controller.}
	\label{lst:controlmode}
\end{listing}

\begin{description}
	\item[Velocity] The default mode when no MIDI clock is locked.
	      A velocity command is sent with the position field set to NaN so the Moteus uses
	      pure velocity control. Acceleration is limited asymmetrically:
	      $1.67\,\text{rev/s}^2$ when spinning up (approximately 4~s to reach tremolo speed)
	      and $2.50\,\text{rev/s}^2$ when spinning down,
	      replicating the different inertia felt during acceleration and deceleration of a
	      mechanical rotor.
	      The integral term is disabled ($k_i\text{-scale} = 0$) to avoid wind-up during
	      speed transitions.

	\item[BeatSync] Entered automatically when \texttt{clockSyncIsLocked()} returns true.
	      The target velocity is set to the smoothed BPM divided by 60,
	      mapping one rotor revolution to one quarter note.
	      The mode starts in a spin-up sub-state using velocity control until the measured
	      speed is within 15\% of the target.
	      Once within tolerance, the tick count and fractional inter-tick phase are used to
	      compute an absolute position setpoint,
	      \begin{equation}
		      \theta_\text{ref}(t) = \frac{n_\text{tick} + \phi}{24} \cdot \delta + \theta_\text{offset},
	      \end{equation}
	      where $n_\text{tick}$ is the MIDI tick count, $\phi \in [0,1)$ is the inter-tick phase
	      from \texttt{clockSyncGetPhase()}, $\delta = 1$ is the subdivision factor
	      (one revolution per quarter note), and $\theta_\text{offset}$ is a fixed offset recorded
	      when the rotor first achieved speed lock.
	      The corresponding code is shown in \autoref{lst:beatsync}.

	      \begin{listing}[H]
		      \begin{minted}{cpp}
uint64_t ticks    = clockSyncGetTickCount();
double   subPhase = clockSyncGetPhase();          // 0..1 within one tick
double   expected = (ticks + subPhase) / 24.0 * SUBDIVISION + s_beatOffset;

cmd.position = expected;   // absolute revolutions
cmd.velocity = targetVel;  // feed-forward: BPM / 60
\end{minted}
		      \caption{Position setpoint computation in BeatSync mode.}
		      \label{lst:beatsync}
	      \end{listing}

	\item[Position] Entered when the reference speed is set to zero and the rotor has
	      decelerated below $0.4\,\text{rev/s}$ ($\approx 24\,\text{RPM}$).
	      The controller snaps to the nearest whole revolution in the direction of travel
	      and holds that position, preventing the rotor from coasting to an arbitrary angle.
\end{description}

\subsection{Midi-listener}
\texttt{midiListnerTask} initialises USB MIDI using the Adafruit TinyUSB library together with
the Arduino MIDI library, then enters a polling loop calling \texttt{MIDI.read()}.
The library dispatches incoming messages to registered callbacks.

Clock callbacks (\texttt{onClock}, \texttt{onStart}, \texttt{onStop}, \texttt{onContinue}, \texttt{onSongPos})
timestamp the event with \texttt{esp\_timer\_get\_time()} immediately and push a \texttt{ClockMsg}
onto the clock queue without blocking, so the callback returns before the next USB frame arrives.

Note-on callbacks recognise three pad notes on MIDI channel~10:
E1 (note~40), F1 (note~41), and F\#1 (note~42) are mapped to \texttt{BUTTON0}, \texttt{BUTTON1},
and \texttt{BUTTON2} respectively, which the input handler translates to chorale speed, stop, and tremolo speed.
Control-change messages on channel~1, controller~7 are forwarded as \texttt{MidiCC} events,
allowing an expression pedal or mod wheel to set rotor speed continuously.

\subsection{Mode-selector}
\texttt{modeSelectorTask} determines which physical control source is active.
A four-position rotary switch is wired as a resistor ladder and read via the ADC on pin~A7.
The raw reading is compared against three thresholds to select one of the four sources
defined by the \texttt{ControlSource} enum in \autoref{lst:controlsource}.

\begin{listing}[H]
	\begin{minted}{cpp}
enum class ControlSource {
    Footswitch,        // two-button footswitch pedal
    ExpressionPedal,   // continuous expression pedal (0-5V)
    MidiKeyboard,      // MIDI note/CC from USB keyboard
    MidiBeatSync       // beat-synchronised mode driven by MIDI clock
};
\end{minted}
	\caption{\texttt{ControlSource} enum defining the four selectable input sources.}
	\label{lst:controlsource}
\end{listing}

A debounce counter requires three consecutive identical ADC readings before committing a change,
preventing glitches from contact bounce or ADC noise.
Switching between footswitch and expression pedal also changes the hardware signal path:
a digital output on pin~D2 drives a SPDT analogue switch, physically routing the TRS socket
to either the footswitch circuit or the expression pedal ADC input.
The active source is stored in a spinlock-protected global read by the input-handler task.

\subsection{Input-handler}
\texttt{inputHandlerTask} sits between raw input events and the \texttt{ReferenceState}
shared with the controller.
The key data structures are shown in \autoref{lst:inputevent}.

\begin{listing}[H]
	\begin{minted}{cpp}
struct ReferenceState {
    float angleDeg;   // target rotor angle (unused in velocity modes)
    float velRPM;     // target rotor speed in RPM
};

enum class InputSource : uint8_t {
    Footswitch, ExpPedal, MidiButton, MidiCC
};

struct InputEvent {
    InputSource source;
    union {
        FootswitchState foot;      // { bool swA, swB }
        ExpPedalState   expPedal;  // { float voltage }
        MidiCCEvent     midiCC;    // { uint8_t value }
        MidiButtonEvent midiButton;
    } data;
};
\end{minted}
	\caption{\texttt{ReferenceState}, \texttt{InputSource}, and \texttt{InputEvent} type definitions.}
	\label{lst:inputevent}
\end{listing}

The task blocks on the input queue and, upon receiving an event, checks the currently selected
control source before acting:

\begin{itemize}
	\item A \texttt{Footswitch} event maps the two-switch state to one of three speeds:
	      switch~A alone selects chorale speed (40~RPM),
	      switches A and B together select tremolo speed (400~RPM),
	      and neither switch sets 0~RPM.
	\item An \texttt{ExpPedal} event maps the raw expression pedal voltage linearly to
	      a speed between 0 and 400~RPM, with a dead-band near zero to prevent creep.
	\item A \texttt{MidiButton} event maps the three pad buttons directly to chorale speed,
	      stop, and tremolo speed.
	\item A \texttt{MidiCC} event maps the 0--127 controller value to a speed between
	      0 and 400~RPM with a small dead-band near zero.
\end{itemize}

The resulting \texttt{ReferenceState} is written atomically via \texttt{referenceSet()},
which uses a spinlock so the controller task always sees a consistent value.
